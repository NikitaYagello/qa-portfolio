# API Testing — практика (Postman)

Отработка ручного тестирования REST API: отправка запросов, проверка статус-кодов и тела ответа, позитивные и негативные сценарии, базовые автотесты в Postman.

**Инструменты:** Postman · JSONPlaceholder · ReqRes

---

## Урок 1 — базовые GET-запросы и чтение ответа

**Объект тестирования:** [JSONPlaceholder](https://jsonplaceholder.typicode.com) (публичный тестовый REST API)

**Что проверял:** корректность ответа на запрос существующего и несуществующего ресурса, базовая проверка статус-кода.

### Проверки

| # | Запрос | Тип | Ожидаемый результат | Факт |
|---|---|---|---|---|
| 1 | `GET /users/1` | Позитивный | 200 OK, в теле — данные пользователя | ✅ 200 |
| 2 | `GET /users` | Позитивный | 200 OK, список пользователей | ✅ 200 |
| 3 | `GET /users/99999` | Негативный | 404 Not Found (ресурса нет) | ✅ 404 |

### Автотест (Postman, вкладка Scripts)
```javascript
pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});
```
Результат: `PASS` (вкладка Test Results).

### Скриншоты
- Успешный запрос (200): `./screenshots/lesson1-200.png`
- Негативный запрос (404): `./screenshots/lesson1-404.png`
- Результаты автотеста: `./screenshots/lesson1-autotest-collection.png`
### Вывод
Освоил базовый цикл проверки API: отправка GET-запроса, чтение статус-кода и тела ответа, разделение позитивного и негативного сценария, базовая ассерция на статус-код.


## Урок 2 — заголовки, авторизация и POST

**Объект тестирования:** [ReqRes](https://reqres.in) (требует заголовок `x-api-key`)

**Что проверял:** авторизацию через заголовок, создание ресурса методом POST, различие статус-кодов 200/201/401/404.

### Проверки

| # | Запрос | Заголовки | Тип | Ожидаемый результат | Факт |
|---|---|---|---|---|---|
| 1 | `GET /api/users?page=2` | без `x-api-key` | Негативный (авторизация) | 401, «Missing API key» | |
| 2 | `GET /api/users?page=2` | `x-api-key` | Позитивный | 200 OK, список + пагинация | |
| 3 | `POST /api/users` + JSON body | `x-api-key`, `Content-Type: application/json` | Позитивный | 201 Created, в ответе `id` и `createdAt` | |
| 4 | `GET /api/users/23` | `x-api-key` | Негативный | 404 Not Found | |

### Тело POST-запроса
```json
{
    "name": "Nikita",
    "job": "QA Engineer"
}
```

### Автотесты
```javascript
pm.test("Status code is 201", function () {
    pm.response.to.have.status(201);
});

pm.test("Response has id", function () {
    pm.expect(pm.response.json()).to.have.property("id");
});
```
Результат: `PASS`.

### Скриншоты
- Без ключа (401): `./screenshots/lesson2-401.png`
- С ключом (200): `./screenshots/lesson2-200.png`
- Создание ресурса (201): `./screenshots/lesson2-201.png`

### Вывод
Разобрал роль заголовков: `x-api-key` отвечает за авторизацию, `Content-Type` — за формат тела. Научился отправлять POST с JSON-телом и проверять создание ресурса. Понимаю разницу между 401 (нет прав) и 404 (ресурс не найден), а также между 200 и 201.


## Урок 3 — коллекция, переменные окружения и запуск

**Цель:** организовать запросы как переиспользуемый рабочий процесс.

**Что сделал:**
- Собрал запросы  в коллекцию `ReqRes API Tests`.
- Вынес повторяющиеся значения в переменные окружения, чтобы не дублировать их в каждом запросе.
- Прогнал всю коллекцию через Collection Runner.
- Экспортировал коллекцию в `collection.json` для переноса/проверки другими.

### Переменные окружения (`ReqRes`)

| Переменная | Значение | Где используется |
|---|---|---|
| `base_url` | `https://reqres.in` | в URL: `{{base_url}}/api/users` |
| `api_key` | `reqres-free-v1` | в заголовке `x-api-key`: `{{api_key}}` |

### Запуск коллекции
Все запросы прогнаны через **Collection Runner** одной кнопкой; в сводке — все тесты `PASS`.

### Скриншоты
- Структура коллекции с переменными: `./screenshots/lesson3-collection.png`
- Сводка прогона (Runner, PASS): `./screenshots/lesson3-runner.png`

### Вывод
Умею организовывать API-проверки в коллекцию, выносить параметры в переменные окружения (быстрое переключение между стендами без правки каждого запроса), запускать набор тестов разом и экспортировать коллекцию для переиспользования. Это базовый, но полноценный рабочий процесс ручного тестирования API.
